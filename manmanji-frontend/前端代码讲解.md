# 一、应用启动流程

```
浏览器输入 URL
       │
       ▼
┌──────────────────────────────────────────┐
│ 1. plugins/01.pinia.ts                   │  创建 Pinia 实例，挂到 Vue 上
│ 2. plugins/auth.client.ts                │  从 localStorage 读 token，恢复到 store
│ 3. middleware/auth.global.ts             │  检查 cookie/Pinia → 有 token？放行 : 跳 /about
│ 4. 页面组件渲染                           │  onMounted() 里调 API 加载数据
└──────────────────────────────────────────┘
```

**记忆要点**：插件按文件名数字顺序加载（`01.pinia` → `auth.client`），中间件在页面渲染前执行，这是 Nuxt 的运行机制。

# 二、认证体系（面试重点）

## 2.1 登录完整链路

```
用户输入账号密码 → 点登录
      │
      ▼
login.vue 调 useAuth().login({ account, password })
      │
      ▼
useAuth.ts 调 api.post('/api/auth/login', data, auth=false)
      │  ↑ auth: false = 此时没有 token，不传 Authorization 头
      ▼
useApi.ts 发 fetch → 后端返回 { accessToken, refreshToken, user, expiresIn }
      │
      ▼
store.setSession(result)  ← 三件事同时做：
  ├── 存 Pinia store（内存，Vue 响应式）→ 驱动 UI（isAuthenticated）
  ├── 存 localStorage（硬盘，mannote-auth key）→ 刷新不丢
  └── 写 cookie（mannote-token，7天，HttpOnly=false）→ 服务端中间件可读
      │
      ▼
navigateTo('/home')  跳转到主页
```

## 2.2 Token 过期自动刷新（无感刷新）

```
发任意 API 请求 → 后端返回 401（token 过期）
      │
      ▼
useApi.ts 拦截 401 → 调 tryRefresh()
      │
      ├── 刷新成功 → 拿到新 accessToken → 重试刚才的请求 → 用户无感知
      │
      └── 刷新失败（refresh 也过期）→ 清除登录态 → navigateTo('/about')
```

**并发 401 单例控制**：如果同时 3 个请求都返回 401，用一个模块级 `refreshPromise` 变量保证只发一次刷新请求，另外两个等同一个 Promise。

## 2.3 路由守卫

```
用户导航到任意页面
      │
      ▼
是 / 、/about 、/login ？ → 是 → 放行
      │ 否
      ▼
服务端（SSR）：读 cookie 有没有 mannote-token
客户端（CSR）：读 Pinia store.isAuthenticated（含过期判断）
      │
      ├── 通过 → 放行
      └── 没通过 → navigateTo('/about')
```

**面试可讲的点**：为什么服务端和客户端各检查一次？Nuxt 中间件在服务端渲染时跑一次，客户端 hydration 后再跑一次。服务端只能读 cookie（没有 localStorage），客户端可以读 Pinia store（能判断过期时间）。两层检查相互兜底。

## 2.4 角色隔离

`middleware/role-guard.ts`：ADMIN 用户不能访问 `/home`（用户界面），普通用户不能访问 `/admin`。两端实现——服务端从 cookie 里解析 JWT payload 拿 role，客户端从 store 拿。

## 2.5 核心文件

| 文件 | 作用 |
|------|------|
| `composables/useApi.ts` | HTTP 层：发请求 + 加 Authorization 头 + 401 自动刷新 |
| `composables/useAuth.ts` | 业务层：login/register/refresh/logout |
| `stores/auth.ts` | 状态层：存 token/user，判断 isAuthenticated/isAdmin |
| `middleware/auth.global.ts` | 路由拦截：未登录 → /about |
| `middleware/role-guard.ts` | 角色隔离：ADMIN ↔ USER |
| `plugins/auth.client.ts` | 启动恢复：localStorage → Pinia |

# 三、API 层三层架构

```
页面组件                   ← 只管展示和交互
    │ 调用
composables/useAuth.ts     ← 业务编排（调 API + 更新 store）
composables/useArticle.ts
    │ 调用
composables/useApi.ts      ← 网络层（fetch + token + 刷新 + 解包）
    │ 调用
后端 Java API              ← /api/*
```

## useApi.ts 每次请求做了什么

```
useApi().post('/api/articles/publish', body)
      │
      ├── 1. 从 localStorage 读 accessToken
      ├── 2. 设请求头：Content-Type: application/json + Authorization: Bearer xxx
      ├── 3. fetch() 发请求
      ├── 4. 解析 JSON → 检查 result.code === 200
      │       ├── 是 200 → 返回 result.data（已解包，调用方拿到的就是业务数据）
      │       └── 是 401 → 自动刷新 token → 重试
      └── 5. 出错抛异常
```

**为什么 API 层自己读 localStorage 而不从 Pinia 读？**

避免循环依赖。useApi 是最底层，如果它 import Pinia，而 Pinia 的 auth store 又依赖 useApi 发登录请求，就形成了循环引用。解决方案是：API 层和 store 各自独立读写 localStorage 的同一个 key（`mannote-auth`），数据一致但代码解耦。

# 四、页面启动流程

## 4.1 index.vue（`/`）—— 智能重定向

```
访问 /
  │
  ├── 服务端：读 cookie 里的 mannote-token → 解析 payload.role
  │     ├── ADMIN → /admin
  │     ├── USER  → /home
  │     └── 无    → /about
  │
  └── 客户端：读 store.isAuthenticated + store.isAdmin
        ├── ADMIN → /admin
        ├── USER  → /home
        └── 无    → /about
```

没有模板内容，纯脚本重定向。设计原因：Nuxt SPA 模式下 `/` 只有一个 `index.html`，如果它包含 landing 内容，nginx fallback 会把所有路由都显示成 landing 页。

## 4.2 about.vue（`/about`）—— Landing 页

展示"慢慢记"品牌介绍，包含：hero 标题、slogan（"专注技术，慢下来思考"）、ICP 备案号、"开始使用"和"了解更多"两个按钮。公开页面，无需登录。

## 4.3 login.vue（`/login`）—— 登录/注册

双标签页切换：
- **登录**：account（用户名或邮箱）+ password
- **注册**：username + email + 验证码 + password + 确认密码
- 验证码通过 `POST /api/auth/send-code` 发送，60 秒倒计时防重复
- 成功跳转：ADMIN → /admin，普通用户 → /home

# 五、Toast 通知系统

## 5.1 工作原理

```
布局层（layouts/editor.vue 或 layouts/blank.vue）
    │ provideToast()  ← 把 show() 函数注入到组件树
    ▼
子组件中调用 injectToast()
    │ toast.show('保存成功', 'success')
    ▼
Toast.vue 渲染通知卡片（顶部居中，3 秒自动消失）
```

## 5.2 实现细节

- **位置**：`position: fixed; top: 16px; left: 50%; transform: translateX(-50%)`，屏幕顶部居中
- **z-index**: 9999（最高层，覆盖一切）
- **类型**：`success`（绿色左边框 + 对勾图标）、`error`（红色左边框 + 叉号图标）
- **动画**：`TransitionGroup`，进入时从上方滑入（`translateY(-12px)` → `0`，`opacity 0 → 1`），离开时反向
- **自动消失**：`setTimeout(() => remove, 3000)`
- **使用场景**：保存成功/失败、图片上传错误、发布错误、标签/分组数量超限

## 5.3 核心技术点

**provide/inject 模式**：不用 Pinia 存 toast 状态，而是用 Vue 的依赖注入。原因是 toast 是纯 UI 层的临时状态，不属于全局业务状态。看起来简单，但面试时能说出"我区分了全局状态（Pinia）和 UI 状态（provide/inject）"是加分项。

# 六、编辑器架构（`/write`）

编辑器是整个项目最复杂的前端模块，下面逐个拆解。

## 6.1 组件树

```
pages/write.vue（编排层：顶栏显隐 + 发布设置）
  │
  ├── EditorTopNav.vue          ← 顶栏（返回 + 标题输入 + 头像下拉）
  ├── EditorView.vue            ← 主卡片（★ 核心）
  │   ├── [工具栏]                ← 格式按钮 + 下拉菜单 + 视图切换
  │   ├── EditorTextarea.vue     ← CodeMirror 6 编辑器（左/独占）
  │   ├── DraggableDivider.vue   ← 拖拽分隔线（仅对比模式）
  │   ├── EditorPreview.vue      ← Markdown 预览面板（右/独占）
  │   └── [链接插入弹窗]          ← 模态弹窗
  ├── RightFloatingPanel.vue    ← 右侧浮动面板（AI / 大纲 / 语法帮助）
  └── BottomStatusBar.vue       ← 底部状态栏（字数 + 保存/发布按钮）
```

## 6.2 状态共享机制（provide/inject）

编辑器的所有组件不通过 Pinia 通信，而是通过 `Symbol('editor')` 做 provide/inject：

```
EditorView.vue
  │ createEditorState(articleId)  → 创建状态对象
  │ provide(EDITOR_KEY, state)    → 注入到子组件树
  ▼
所有子组件
  │ const editor = inject(EDITOR_KEY)  → 获取状态
  │ editor.content    ← 文章正文
  │ editor.title      ← 标题
  │ editor.viewMode   ← 当前视图（edit/split/preview）
  │ editor.save()     ← 保存草稿
  │ editor.publish()  ← 发���文章
  │ editor.isSaving   ← 保存中？
  │ editor.lastSavedAt ← 上次保存时间
```

**为什么不用 Pinia？** 编辑器状态是页面级的、临时的（关掉页面就不需要了），Pinia 更适合全局持久状态（如登录态）。这是"全局状态 vs 页面状态"的经典区分。

## 6.3 三种视图模式

```
edit          ┌─────────────────────────────────────┐
              │          CodeMirror 编辑器           │
              └─────────────────────────────────────┘

split         ┌───────────┬──┬──────────────────────┐
              │  编辑器    │拖│   Markdown 预览        │
              └───────────┴──┴──────────────────────┘

preview       ┌─────────────────────────────────────┐
              │         Markdown 预览                │
              └─────────────────────────────────────┘
```

工具栏中央的切换按钮（编辑 | 对比 | 预览），`viewMode` 值驱动 v-if 条件渲染。

## 6.4 可拖拽分隔线（DraggableDivider.vue）

- 16px 宽的可点击区域，在对比模式下调整编辑区和预览区的宽度比例
- 拖拽时用 `requestAnimationFrame` 更新 CSS 变量 `--panel-px`，GPU 加速（`will-change: width` + `translateZ(0)`）
- 双击恢复 50/50
- 右面板最小 250px

**面试可讲**：为什么用 rAF？拖拽时鼠标每移动 1px 可能触发几十次事件，直接改 DOM 会卡。rAF 把更新合并到浏览器下一次重绘，60fps 不掉帧。

## 6.5 滚动同步（双向）

编辑器滚动 → 预览跟着滚，预览滚动 → 编辑器跟着滚。用两个 guard flag 防循环（`isTextareaDriven` / `isPreviewDriven`）。

```
编辑器滚动事件
  → 算出滚动比例 = scrollTop / (scrollHeight - clientHeight)
  → 预览区 set scrollTop = 比例 × (previewScrollHeight - previewClientHeight)
  → guard flag 防止预览区的 scroll 事件反过来再触发编辑器滚动
```

## 6.6 CodeMirror 6 集成（EditorTextarea.vue）

- 使用 `@codemirror/view` + `@codemirror/state` + `@codemirror/lang-markdown` + `@codemirror/commands`
- 自定义主题：JetBrains Mono 16px/1.75，`.cm-line` padding 设为 0（确保与预览区文字宽度一致）
- 内容同步：`updateListener` 检测 `docChanged` → 更新外部 `content` ref；外部变化 → `view.dispatch()` 替换全文
- `insertAtCursor(text)`：在光标位置插入文本（加粗、链接、图片等工具栏功能都调用它）

## 6.7 Markdown 渲染管线（useMarkdownRenderer.ts）

```
原始 Markdown 文本
      │
      ▼
markdown-it 实例（配置了 linkify + typographer + breaks）
      │
      ├── 插件：emoji / KaTeX（数学公式）/ task-lists / footnote / mark
      │
      ├── 代码块：highlight.js 语法高亮 + 语言标签 + 复制按钮
      │          Mermaid 图表渲染
      │
      ├── 链接：target="_blank" + rel="noopener noreferrer"
      │
      ├── 标题：自动注入 id="heading-0", id="heading-1", ...
      │
      ▼
渲染后的 HTML → v-html 插入 DOM
      │
      ▼
解析 <h1>~<h6> 标签 → 生成 TocItem[] → 驱动右侧大纲
```

**异步插件加载**：markdown-it-katex 等 CJS 插件通过动态 `import()` 加载，加载完成后触发重新渲染。这是因为 Vite ESM 环境对 CJS 模块的处理需要异步。

## 6.8 图片上传

```
点工具栏图片按钮 → 隐藏的 <input type="file"> 打开
  → 选文件 → useArticle().uploadImage(file)
  → FormData multipart 上传到 /api/storage/image
  → 后端返回图片 URL
  → insertAtCursor('![描述](图片URL)')
  → Toast 提示上传结果
```

封面图上传逻辑类似，但用 `URL.createObjectURL(file)` 生成本地预览，无需上传即可预览。

## 6.9 发布设置（write.vue）

编辑器下方有发布设置区域：

| 设置项 | 实现 |

| **分组选择器** | 同上布局，按 Enter 可创建新分组，异步调 `createGroup()` 写入后端 |

## 6.10 链接插入弹窗（EditorView.vue）

# 十、TOC 滚动高亮追踪（useTocActive.ts）

**作用**：右侧大纲目录自动高亮当前正在阅读的标题。

**实现原理**：`IntersectionObserver`
- 给每个标题元素加观察器
- `rootMargin: '0px 0px -70% 0px'` → 标题进入视口顶部 30% 区域时才算"当前标题"
- 找到第一个可见标题 → 更新 `activeTocId`
- `tocItems` 变化时（文章切换），重新绑定观察器

**为什么用 IntersectionObserver 而不是监听 scroll 事件？**
scroll 事件频率太高，自己做节流会影响精度。IntersectionObserver 是浏览器原生 API，异步触发，不阻塞主线程。

# 十三、面试速查表

## 高频问题 + 你的回答

| 问题 | 回答要点 |
|------|---------|
| **跨域怎么处理？** | 开发环境 Nuxt devProxy 代理 `/api/*` → `localhost:8080`，生产环境 nginx 反代，前后端同域，不存在跨域 |
| **Token 存哪里？** | 三个地方：localStorage（API 层读写）、Pinia store（组件用）、cookie（服务端中间件用） |
| **Token 过期怎么办？** | `useApi.ts` 拦截 401，用 refresh token 换新 access token，重试原请求。并发请求用单例 Promise 防重复刷新 |
| **路由守卫怎么做的？** | Nuxt 全局中间件 `auth.global.ts`，白名单 `/` `/about` `/login`，其余检查 cookie/Pinia，不过就跳 `/about` |
| **前端状态怎么分层的？** | Pinia 存全局持久状态（登录态），provide/inject 存页面临时状态（编辑器），普通 ref 存组件内部状态 |
| **编辑器怎么实现的？** | CodeMirror 6 + markdown-it，双向滚动同步（比例法 + guard flag），provide/inject 共享状态 |
| **TOC 高亮怎么做的？** | IntersectionObserver，rootMargin -70%，标题进入视口顶部 30% 时高亮 |
| **为什么 API 层不依赖 Pinia？** | 避免循环引用。API 层自己读 localStorage，和 store 读写同一个 key 但互不依赖 |
| **未保存怎么提示的？** | `onBeforeRouteLeave`（路由跳转）+ `beforeunload`（关闭标签页），弹出 confirm 对话框 |
